# Team Working Agreement

**Course:** CS 691 — Computer Science Capstone Project, Spring 2026  
**Team:** Group 4 — AI Interior Designer v2  
**Established:** January 2026

---

## Purpose

This working agreement defines how Group 4 operates as a team. All members have agreed to and are accountable to these norms. This document is reviewed at each sprint retrospective.

---

## Meeting Schedule

| Meeting Type | Frequency | Day / Time | Platform |
|-------------|-----------|------------|----------|
| Stand-up | 2× per week | Tuesday & Thursday, 7:00 PM EST | Discord |
| Sprint Planning | Start of each sprint | Monday, 6:00 PM EST | Zoom |
| Sprint Review / Demo | End of each sprint | Friday, 6:00 PM EST | Zoom |
| Sprint Retrospective | After each review | Same Friday, ~7:00 PM EST | Zoom |

**Attendance Policy:**
- Members must notify the team at least **2 hours in advance** if they cannot attend
- Missing more than 2 stand-ups per sprint without notice is flagged to the instructor
- All Zoom meetings are recorded for members who miss them

---

## Communication

- **Primary channel:** Discord server (Group 4 — AI Interior Designer)
- **Response time:** Reply to Discord messages within **24 hours** on weekdays
- **Urgent issues:** Tag `@everyone` in Discord or text directly
- **File sharing:** GitHub repository only — no emailing code
- **Meeting notes:** Rotating responsibility; posted to Discord #meeting-notes within 24 hours

---

## Work Distribution & Deadlines

- Tasks are assigned during sprint planning and tracked on the **GitHub Projects board**
- Each member estimates their own story points
- Work must be **completed 24 hours before the sprint demo**, not the night before
- No task should be handed off to one person an hour before a deadline
- If you are blocked, post in Discord immediately — do not wait for the next stand-up
- Pull requests require **at least 1 reviewer approval** before merging to `main`

---

## Accountability

| Situation | What We Do |
|-----------|-----------|
| Can't finish task on time | Post in Discord ASAP, ask for help or reassignment |
| Missed a meeting | Watch the recording, catch up on notes within 24 hours |
| Conflict with a teammate | Resolve directly first; escalate to PM if unresolved |
| A bug blocks everyone | Drop current work and fix the blocker together |
| Member consistently misses commitments | PM addresses it; escalate to instructor if needed |

---

## Code Standards

- All code pushed to feature branches — never push directly to `main`
- Branch naming: `feature/description`, `fix/description`, `docs/description`
- Commit messages must be descriptive (e.g., `fix: resolve CORS error on /generate`)
- No commented-out code committed to `main`
- Every PR must include a description of what changed and why

---

## Definition of Done

A task is **Done** when:
- [ ] Code is written and tested locally
- [ ] PR is opened, reviewed, and approved by at least 1 teammate
- [ ] Merged to `main` without breaking existing features
- [ ] Documentation updated if the feature changes user-facing behavior
- [ ] Demoed to the team at sprint review
